# DiTing 后续整体开发路线

更新时间：2026-07-17

## 1. 总体原则

DiTing 后续开发不能只停留在“操作日志查询系统”。短期要把 Linux 主机审计做扎实，中长期要充分利用 Tetragon 的内核级观测和安全能力，逐步形成主机行为审计、风险分析、实时告警和安全处置平台。

后续开发遵循三条主线：

- 主线 A：主机审计优先做完整，围绕 Linux 主机、用户、命令、风险事件形成闭环。
- 主线 B：逐步接入 Tetragon gRPC stream 和更多事件类型，不能长期只依赖落盘 JSON。
- 主线 C：从日志查询走向安全分析，围绕风险识别、行为关联、规则命中、处置闭环持续演进。

近期不要把主要精力放在 K8s 页面能力上。Namespace、Pod、容器上下文可以保留在底层数据和详情中，但当前产品主线以主机审计为主。

## 2. 阶段一：主机审计闭环

目标：让 DiTing 在真实 Linux 主机审计场景下可用、好用、可排查。

重点功能：

- 主机审计列表
  - 按稳定 `host_id` 优先聚合主机。
  - 展示主机名、host_id、node_name。
  - 展示命令数、活跃用户数、高危事件数、首次活动、最近活动。
  - 支持主机搜索。
  - 支持导出主机列表。

- 主机详情
  - 主机概览。
  - 高危命令列表。
  - 用户分布。
  - 命令明细。
  - 命令明细筛选：用户、命令关键字、风险等级。
  - 命令明细支持服务端分页，避免只查前 100 条。
  - 支持导出当前主机命令明细。

- 主机画像增强
  - TOP 命令。
  - TOP 进程。
  - 高危规则命中分布。
  - 最近活跃用户。
  - 最近高危操作时间线。

## 3. 阶段二：命令审计按主机补齐

目标：命令审计不仅能看“执行了什么命令”，还要能看“在哪台主机、由谁执行、风险如何”。

重点功能：

- 命令审计列表增加主机维度。
- 展示最近执行主机、涉及主机数。
- 支持主机筛选。
- 支持登录用户、执行用户筛选。
- 支持风险等级筛选。
- 命令详情展示：
  - 该命令在哪些主机执行过。
  - 用户分布。
  - 首次执行、最近执行。
  - 高危命中情况。

当前阶段不优先做 Namespace / Pod 维度展开，避免偏离主机审计主线。

## 4. 阶段三：用户审计按主机补齐

目标：围绕用户行为形成“用户 -> 主机 -> 命令 -> 风险”的审计视图。

重点功能：

- 用户审计列表增强
  - 命令数。
  - 涉及主机数。
  - 高危事件数。
  - 首次活动、最近活动。

- 用户详情
  - 该用户在哪些主机执行过命令。
  - 该用户 TOP 命令。
  - 该用户高危命令。
  - 该用户完整命令明细。

- 用户审计筛选
  - 主机。
  - 命令关键字。
  - 风险等级。
  - 时间范围。

## 5. 阶段四：Tetragon gRPC Stream 接入

目标：不能长期只靠 Tetragon JSON 文件落盘采集。需要利用 Tetragon gRPC stream 形成实时采集能力。

重点功能：

- Collector 支持 `input_mode: grpc`。
- 连接 Tetragon gRPC API。
- 实时接收 `ProcessExec`、`ProcessExit`。
- 保留 file tail 作为兼容模式。
- gRPC 断线重连。
- gRPC 采集心跳。
- 采集状态页展示 gRPC 连接状态、最近事件时间、断线次数、最近错误。

设计原则：

- file 模式继续可用，方便测试和低门槛部署。
- grpc 模式作为生产实时采集方向。
- 采集链路必须能定位延迟、断线、写入失败等问题。

## 6. 阶段五：Tetragon 事件类型扩展

目标：充分利用 Tetragon 的 eBPF 能力，不只采集进程执行。

优先事件类型：

- 文件访问事件
  - 敏感文件读取。
  - 敏感文件写入。
  - 权限变更。
  - 删除行为。

- 网络连接事件
  - 外联 IP。
  - 端口。
  - 进程关联。
  - 用户关联。

- DNS 事件
  - 解析域名。
  - 可疑域名关联。
  - 域名与进程、用户、主机的关联分析。

- 进程树增强
  - 父子进程链路。
  - 命令执行链。
  - 可疑进程 ancestry 分析。

K8s / 容器上下文可作为事件上下文保留，但不作为近期页面主线。

## 7. 阶段六：主机安全行为分析

目标：从单条日志查询升级为主机风险画像和行为关联。

重点能力：

- 主机风险画像
  - 高危命令。
  - 敏感文件访问。
  - 异常外联。
  - 规则命中趋势。
  - 活跃用户变化。

- 用户行为链路
  - 登录用户。
  - 执行用户。
  - 命令。
  - 文件行为。
  - 网络行为。

- 风险聚合
  - 同一主机同一时间段多条风险合并成安全事件。
  - 同一用户跨主机行为关联。
  - 同一命令跨主机扩散分析。

## 8. 阶段七：规则与风险分析

目标：规则系统不只是命令行匹配，而是覆盖 Tetragon 多事件、多字段、多行为链。

重点功能：

- 规则字段扩展
  - 命令行。
  - 进程名。
  - 用户。
  - 主机。
  - 文件路径。
  - 网络目标。
  - DNS 域名。
  - 父进程。

- 规则分析
  - 命中最多的规则。
  - 长期未命中的规则。
  - 规则命中的主机数、用户数、命令数。
  - 规则误报处置统计。

- 规则测试增强
  - 从真实事件一键带入测试。
  - 查看未命中的条件。
  - 支持事件回放测试。

- 规则模板
  - 主机入侵。
  - 提权。
  - 数据泄露。
  - 横向移动。
  - 异常外联。

## 9. 阶段八：实时告警与处置

目标：结合 gRPC stream 实时采集能力，形成实时风险告警。

重点功能：

- 前端实时推送
  - WebSocket 或 SSE。
  - 新高危事件提醒。
  - 主机离线提醒。
  - Collector 异常提醒。

- 告警通知
  - 企业微信。
  - 钉钉。
  - 邮件。

- 风险处置流程
  - 未处理。
  - 已确认。
  - 已忽略。
  - 备注。
  - 处理人。
  - 处理时间。
  - 处置历史。

## 10. 阶段九：主机资产整理

目标：主机资产围绕真实 Linux 主机建模，不以 K8s node/displayName 为核心。

主机资产模型：

- host_id。
- 主机名。
- IP。
- 环境。
- 负责人。
- 部门。
- 备注。

联动能力：

- 主机审计页展示负责人、环境、IP。
- 支持按环境、负责人筛选。
- 主机详情展示资产信息。
- 提示未配置稳定 host_id 的主机。

## 11. 阶段十：生产化采集架构

目标：降低数据库暴露面，提高大规模部署能力。

演进方向：

```text
Collector -> API Ingest -> ClickHouse
```

或：

```text
Collector -> Kafka/NATS -> 后端消费 -> ClickHouse
```

重点功能：

- Collector token。
- API 写入鉴权。
- 数据库账号权限收敛。
- Collector 版本管理。
- Collector 在线状态。
- 采集延迟。
- 错误统计。
- 批量写入与重试。

## 12. 开发顺序建议

近期顺序：

1. 主机审计导出和服务端分页。
2. 命令审计补主机列和主机筛选。
3. 用户审计补主机维度。
4. 规则命中分析。
5. 主机资产按 `host_id` 重整。
6. 采集状态增强。
7. Tetragon gRPC stream 设计与接入。

中长期顺序：

1. 文件访问事件。
2. 网络连接事件。
3. DNS 事件。
4. 主机安全行为分析。
5. 实时告警。
6. API ingest 生产化改造。

## 13. 明确不要偏离的点

- 不要把 DiTing 做成单纯的日志表格查询系统。
- 不要长期只围绕一个页面或一个表格死抠。
- 不要在当前阶段把 K8s 页面能力作为主线。
- 不要忽略 Tetragon gRPC stream 和更多内核事件类型。
- 不要只做配置文件能力，面向运维的配置应尽量在前端可管理。
- 不要牺牲主机审计主线的一致性：主机、用户、命令、风险、规则、采集状态要能串起来。
