# DiTing 下阶段开发计划与防跑偏检查

更新时间：2026-07-21

## 1. 下阶段核心目标

下阶段开发目标不是继续扩展普通日志表格，也不是围绕规则页面做细枝末节优化，而是把 DiTing 从“能看到审计事件”推进到“能围绕 Linux 主机看清行为、风险和关联”。

主线保持三点：

- 主机审计优先：所有新功能优先服务于“哪台主机、哪个用户、什么行为、什么风险、如何追溯”。
- 充分利用 Tetragon：继续发挥 gRPC stream、文件访问、网络连接、进程链和后续 enforcement 能力。
- 风险分析闭环：风险不是单条日志标签，而要能在主机、用户、命令、文件、网络之间串起来。

## 2. 近期必须聚焦的功能

### 2.1 主机风险画像增强

目标：主机详情页成为排查一台 Linux 主机风险的主要入口。

必须包含：

- 主机概览：命令数、活跃用户、高危事件、最近活动。
- 高危命令：命令、登录用户、执行用户、时间、命中规则。
- 敏感文件访问：文件路径、操作、进程、用户、时间、命中规则。
- 网络连接风险：目标 IP、端口、协议、进程、用户、时间、命中规则。
- 最近风险时间线：按时间展示命令、文件、网络风险。
- 规则命中分布：该主机命中了哪些规则，各多少次。

不做：

- 不做 DNS 域名关联，当前以 IP/端口为主。
- 不做 K8s namespace/pod 维度展开。
- 不做负责人、部门等资产管理细节。

### 2.2 文件访问事件补齐

目标：文件审计不只看 `/etc/passwd`，要覆盖 Tetragon 文件能力中的关键行为。

优先事件：

- 敏感文件读取：`/etc/passwd`、`/etc/shadow`、sudoers、SSH 配置和密钥。
- 敏感文件写入：配置文件、账号文件、SSH key、计划任务等。
- 权限变更：`chmod`、`chown`、扩展属性变化。
- 删除行为：`unlink`、`unlinkat`、`rmdir`。

页面落点：

- 主机风险画像。
- 风险事件页。
- 审计调试页只用于排障，不作为主要产品入口。

### 2.3 网络连接风险补齐

目标：网络事件能回答“哪个主机、哪个用户、哪个进程连了哪个 IP/端口”。

优先能力：

- 主机详情网络 Top 可点击查看连接明细。
- 风险识别覆盖高危端口、解释器外联、下载工具外联。
- 网络连接明细展示命令、进程、用户、目标 IP、端口、协议。
- 风险事件页能直接展示网络目标。

不做：

- 不做域名解析依赖。
- 不做外网情报接口。
- 不做全局复杂流量分析。

### 2.4 可疑进程链路

目标：从单条命令升级到行为链，例如 `sshd -> bash -> curl`。

优先能力：

- 事件详情展示父进程链路。
- 主机详情展示可疑进程链。
- 风险事件详情能看到父进程、子进程、命令链。
- 对典型链路做风险标注：
  - `sshd -> bash -> curl/wget`
  - `nginx/apache -> sh/bash`
  - `bash/sh -> python/perl/php -> network_connect`

### 2.5 Tetragon 危险命令拦截规划

Tetragon enforcement / 拦截能力是中后期方向，但不能马上直接上线阻断。

正确顺序：

1. 审计模式稳定，规则误报可控。
2. 风险事件可解释，能看清命中原因和行为链。
3. 增加规则级“只审计 / 可阻断”模式。
4. 增加主机级灰度开关。
5. 增加白名单和紧急关闭开关。
6. 小范围启用拦截。

短期只做设计准备，不直接做阻断下发。

## 3. 中期功能

### 3.1 实时告警

目标：结合 gRPC stream，把高危事件从查询式页面变成实时提醒。

优先能力：

- WebSocket 或 SSE 推送新高危事件。
- 主机离线提醒。
- Collector 异常提醒。
- 前端通知入口。

暂不做：

- 企业微信、钉钉、邮件等外部通知可后置。

### 3.2 处置闭环

目标：风险事件能被确认、忽略、备注和追踪。

已有状态可继续增强：

- 未处理。
- 已确认。
- 已忽略。
- 备注。
- 处理人。
- 处理时间。

暂不做复杂工单系统。

### 3.3 生产化采集架构

当前 Collector 直连 PostgreSQL 和 ClickHouse，开发效率高但生产风险较大。

中期方向：

```text
Collector -> API Ingest -> ClickHouse
```

或：

```text
Collector -> Kafka/NATS -> 后端消费 -> ClickHouse
```

优先准备：

- Collector token。
- API ingest endpoint。
- Collector 版本和在线状态。
- 数据库权限收敛。

## 4. 明确暂不开发

以下内容容易跑偏，近期不开发：

- K8s 专用页面、namespace/pod 分析主线。
- DNS 域名审计和外部威胁情报。
- 规则历史回放和历史数据回填。
- 复杂规则未命中解释器。
- 资产负责人、部门、环境等管理细节。
- 报表自动生成。
- 多租户。
- SIEM/SOAR 对接。

这些不是不要，而是等主机审计闭环稳定后再做。

## 5. 每次新功能开发前检查表

每次开始新功能前必须先回答：

1. 这个功能是否服务于 Linux 主机审计主线？
2. 它是否能让“主机 -> 用户 -> 命令/文件/网络 -> 风险 -> 明细”更完整？
3. 它是否利用或强化了 Tetragon 能力？
4. 它是否会把精力带到 K8s、资产字段、规则页面细节或历史回填上？
5. 它是否有明确页面落点，而不是只增加后端字段？
6. 它是否能用真实数据验证？
7. 它是否避免手工 SQL，所需表和默认数据是否能启动或迁移自动创建？

如果第 1、2、3 条不能回答“是”，先不要开发。

## 6. 开发优先级建议

近期顺序：

1. 主机风险画像增强。
2. 文件访问事件补齐：读取、写入、权限变更、删除。
3. 网络风险在主机画像中的关联增强。
4. 可疑进程链路。
5. 实时高危事件推送。
6. 处置闭环增强。
7. Tetragon enforcement 拦截能力设计。

## 7. 当前已完成能力与下一步关系

已经具备：

- Tetragon gRPC stream 接入。
- process_exec/process_exit 采集。
- file_access/network_connect 初步采集。
- 主机审计、命令审计、用户审计基础页面。
- 主机详情中已有文件和网络画像雏形。
- 风险事件可显示进程、文件、网络风险。
- 规则支持命令、文件、网络字段。

下一步不要再继续扩规则页面细节，而是把这些能力合并进主机风险画像和行为链路。

## 8. 质量要求

- 新功能必须有后端测试。
- 前端必须通过 `npm run build`。
- 后端必须通过 `go test ./...`。
- 所有需要初始化的 PostgreSQL 表和默认数据必须通过 bootstrap 或 migration 自动创建。
- 不允许要求用户手工执行零散 SQL。
- 对真实数据页面，必须考虑分页、时间范围、主机筛选，避免大范围全表查询。
